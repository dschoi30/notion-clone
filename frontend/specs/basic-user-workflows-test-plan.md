## 개요

### 문서 정보
- **프로젝트명**: Notion Clone
- **작성일**: 2026-01-10
- **작성자**: QA Team
- **버전**: 1.0

### 테스트 목적
본 문서는 Notion Clone 애플리케이션의 기본 사용자 워크플로우에 대한 포괄적인 테스트 시나리오를 정의합니다. 주요 목적은 다음과 같습니다:
- 핵심 기능의 정상 동작 검증
- 사용자 경험의 일관성 확인
- Edge case 및 오류 처리 검증
- 회귀 테스트를 위한 기준 마련

### 테스트 환경
- **Frontend URL**: http://localhost:5173
- **Backend API**: http://localhost:8080
- **브라우저**: Chrome 최신 버전, Firefox 최신 버전
- **전제 조건**:
  - Backend 서버가 실행 중이어야 함
  - PostgreSQL 데이터베이스가 정상 작동해야 함
  - 테스트 시작 전 깨끗한 상태 (fresh state)

## 테스트 시나리오

## 1. 인증 플로우

### 1.1 회원가입 - 정상 케이스

**목적**: 신규 사용자가 정상적으로 회원가입할 수 있는지 확인

**전제 조건**:
- 애플리케이션이 로그아웃 상태
- 테스트 이메일이 데이터베이스에 존재하지 않음

**테스트 단계**:
1. 브라우저에서 `http://localhost:5173/register` 접속
2. 회원가입 페이지가 표시되는지 확인
3. 이름 입력 필드에 "테스트 사용자" 입력
4. 이메일 입력 필드에 고유한 이메일 입력 (예: `test_${timestamp}@example.com`)
5. 비밀번호 입력 필드에 "password123" 입력
6. "회원가입" 버튼 클릭

**예상 결과**:
- 회원가입 성공 후 자동으로 로그인됨
- 메인 페이지(루트 또는 문서 페이지)로 리다이렉트
- 사이드바가 표시됨
- 워크스페이스가 자동 생성됨 ("테스트 사용자의 워크스페이스")

**성공 기준**:
- [ ] URL이 `/` 또는 `/{documentId}-{slug}` 형태로 변경됨
- [ ] 사이드바(`<aside>`)가 화면에 표시됨
- [ ] 워크스페이스 이름이 정확히 표시됨

---

### 1.2 회원가입 - 중복 이메일

**목적**: 이미 등록된 이메일로 회원가입 시도 시 적절한 오류 처리 확인

**전제 조건**:
- 이메일 "existing@example.com"이 이미 등록되어 있음

**테스트 단계**:
1. `/register` 페이지 접속
2. 이름: "새 사용자" 입력
3. 이메일: "existing@example.com" 입력
4. 비밀번호: "password123" 입력
5. "회원가입" 버튼 클릭

**예상 결과**:
- 회원가입 실패
- 오류 메시지 또는 토스트 알림 표시
- 회원가입 페이지에 그대로 머물러 있음

**성공 기준**:
- [ ] URL이 `/register`에 유지됨
- [ ] 사용자에게 오류가 명확히 전달됨

---

### 1.3 로그인 - 정상 케이스

**목적**: 등록된 사용자가 정상적으로 로그인할 수 있는지 확인

**전제 조건**:
- 이메일 "test@example.com", 비밀번호 "password123"로 등록된 사용자 존재
- 로그아웃 상태

**테스트 단계**:
1. `/login` 페이지 접속
2. 이메일 필드에 "test@example.com" 입력
3. 비밀번호 필드에 "password123" 입력
4. "로그인" 버튼 클릭

**예상 결과**:
- 로그인 성공
- 메인 페이지로 리다이렉트
- 사이드바가 표시됨
- 사용자의 워크스페이스가 표시됨

**성공 기준**:
- [ ] URL이 루트 또는 문서 페이지로 변경됨
- [ ] 사이드바가 정상적으로 렌더링됨
- [ ] 로컬 스토리지에 JWT 토큰이 저장됨

---

### 1.4 로그인 - 잘못된 자격증명

**목적**: 잘못된 이메일/비밀번호로 로그인 시도 시 적절한 오류 처리 확인

**테스트 단계**:
1. `/login` 페이지 접속
2. 이메일: "wrong@example.com" 입력
3. 비밀번호: "wrongpassword" 입력
4. "로그인" 버튼 클릭

**예상 결과**:
- 로그인 실패
- 로그인 페이지에 그대로 머물러 있음
- 로그인 폼이 여전히 표시됨

**성공 기준**:
- [ ] URL이 `/login`에 유지됨
- [ ] 이메일 입력 필드가 여전히 표시됨
- [ ] 로그인 버튼이 여전히 표시됨

---

### 1.5 로그아웃

**목적**: 로그인된 사용자가 정상적으로 로그아웃할 수 있는지 확인

**전제 조건**:
- 로그인된 상태

**테스트 단계**:
1. 메인 페이지에서 사이드바 확인
2. 워크스페이스 헤더(`data-testid="workspace-header"`) 클릭
3. 드롭다운 메뉴가 열리는지 확인
4. "로그아웃" 버튼(`data-testid="logout-button"`) 클릭

**예상 결과**:
- 로그아웃 성공
- 로그인 페이지로 리다이렉트
- 로그인 폼이 표시됨
- 로컬 스토리지에서 토큰 제거됨

**성공 기준**:
- [ ] URL이 `/login`으로 변경됨
- [ ] 로그인 폼이 표시됨
- [ ] 사이드바가 표시되지 않음

---

## 2. 워크스페이스 관리

### 2.1 워크스페이스 표시

**목적**: 로그인 후 사용자의 워크스페이스가 정상적으로 표시되는지 확인

**전제 조건**:
- 로그인된 상태

**테스트 단계**:
1. 메인 페이지 접속
2. 사이드바 상단의 워크스페이스 영역 확인

**예상 결과**:
- 워크스페이스 이름이 표시됨 (예: "{사용자명}의 워크스페이스")
- 워크스페이스 아이콘이 표시됨

**성공 기준**:
- [ ] 워크스페이스 헤더가 표시됨
- [ ] 워크스페이스 이름이 정확함

---

### 2.2 워크스페이스 설정 접근

**목적**: 워크스페이스 설정 모달에 접근할 수 있는지 확인

**전제 조건**:
- 로그인된 상태

**테스트 단계**:
1. 워크스페이스 헤더 클릭
2. 드롭다운 메뉴에서 "설정" 또는 "워크스페이스 설정" 옵션 확인
3. 설정 옵션 클릭

**예상 결과**:
- 워크스페이스 설정 모달이 열림
- 워크스페이스 이름 수정 가능
- 기타 설정 옵션이 표시됨

**성공 기준**:
- [ ] 모달이 화면에 표시됨
- [ ] 워크스페이스 이름 입력 필드가 현재 값으로 채워져 있음

---

## 3. 문서 생성 및 관리

### 3.1 새 문서 생성

**목적**: 사용자가 새 문서를 생성할 수 있는지 확인

**전제 조건**:
- 로그인된 상태
- 워크스페이스가 존재함

**테스트 단계**:
1. 사이드바에서 "새 문서" 버튼 클릭
2. 제목 입력 필드가 나타나는지 확인
3. 제목: "테스트 문서" 입력
4. Enter 키 입력

**예상 결과**:
- 새 문서가 생성됨
- 문서 페이지로 자동 이동
- URL이 `/{documentId}-test-document` 형태로 변경됨
- 사이드바에 새 문서가 추가됨

**성공 기준**:
- [ ] URL에 문서 ID가 포함됨
- [ ] 문서 제목이 정확히 표시됨
- [ ] 사이드바 문서 목록에 새 문서가 표시됨

---

### 3.2 문서 제목 수정

**목적**: 문서 제목을 수정할 수 있는지 확인

**전제 조건**:
- 문서가 열려 있음

**테스트 단계**:
1. 문서 페이지 상단의 제목 영역 클릭
2. 제목을 "수정된 문서 제목"으로 변경
3. Enter 키 입력 또는 다른 영역 클릭

**예상 결과**:
- 제목이 즉시 업데이트됨
- 사이드바의 문서 이름도 업데이트됨
- URL의 slug가 변경됨 (선택적)

**성공 기준**:
- [ ] 제목이 화면에 즉시 반영됨
- [ ] 사이드바의 문서 이름이 동기화됨

---

### 3.3 문서 선택 및 전환

**목적**: 사이드바에서 다른 문서를 선택하여 전환할 수 있는지 확인

**전제 조건**:
- 최소 2개 이상의 문서가 존재함

**테스트 단계**:
1. 사이드바에서 첫 번째 문서 클릭
2. 해당 문서가 열리는지 확인
3. 사이드바에서 두 번째 문서 클릭
4. 두 번째 문서로 전환되는지 확인

**예상 결과**:
- 클릭한 문서가 즉시 로드됨
- URL이 해당 문서로 변경됨
- 사이드바에서 현재 문서가 하이라이트됨

**성공 기준**:
- [ ] URL이 정확히 변경됨
- [ ] 문서 내용이 정확히 로드됨
- [ ] 사이드바에서 선택된 문서가 시각적으로 구분됨

---

### 3.4 문서 삭제

**목적**: 문서를 휴지통으로 이동할 수 있는지 확인

**전제 조건**:
- 삭제할 문서가 존재함

**테스트 단계**:
1. 사이드바에서 문서 위에 마우스 호버
2. 휴지통 아이콘(TrashIcon) 클릭
3. 확인 다이얼로그가 나타나면 "삭제" 확인

**예상 결과**:
- 문서가 사이드바 목록에서 사라짐
- 다른 문서로 자동 이동 (사용 가능한 경우)
- 문서가 휴지통으로 이동됨

**성공 기준**:
- [ ] 사이드바에서 문서가 제거됨
- [ ] 휴지통에서 삭제된 문서를 찾을 수 있음

---

## 4. 문서 편집 (페이지 뷰)

### 4.1 문서 내용 작성

**목적**: TipTap 에디터를 사용하여 문서 내용을 작성할 수 있는지 확인

**전제 조건**:
- 문서가 페이지 뷰 모드로 열려 있음

**테스트 단계**:
1. 에디터 영역 클릭하여 포커스
2. "첫 번째 단락입니다." 입력
3. Enter 키 2번 입력
4. "두 번째 단락입니다." 입력

**예상 결과**:
- 입력한 텍스트가 즉시 표시됨
- 단락이 구분되어 표시됨
- 내용이 자동 저장됨 (실시간 저장)

**성공 기준**:
- [ ] 텍스트가 정확히 입력됨
- [ ] 단락 구분이 올바름
- [ ] 페이지 새로고침 후에도 내용이 유지됨

---

### 4.2 텍스트 서식 적용

**목적**: 텍스트에 기본 서식(볼드, 이탤릭 등)을 적용할 수 있는지 확인

**전제 조건**:
- 에디터에 텍스트가 입력되어 있음

**테스트 단계**:
1. 텍스트 일부 선택
2. 에디터 툴바에서 볼드(B) 버튼 클릭
3. 다른 텍스트 선택
4. 이탤릭(I) 버튼 클릭

**예상 결과**:
- 선택한 텍스트에 서식이 적용됨
- 툴바에서 활성화된 서식이 시각적으로 표시됨

**성공 기준**:
- [ ] 볼드 텍스트가 굵게 표시됨
- [ ] 이탤릭 텍스트가 기울임꼴로 표시됨
- [ ] 서식이 저장되어 유지됨

---

### 4.3 목록 생성

**목적**: 순서 있는 목록과 순서 없는 목록을 생성할 수 있는지 확인

**테스트 단계**:
1. 에디터에서 새 줄 시작
2. "- " (대시 공백) 입력
3. "첫 번째 항목" 입력 후 Enter
4. "두 번째 항목" 입력

**예상 결과**:
- 순서 없는 목록(bullet list)이 자동 생성됨
- Enter 키로 새 항목이 추가됨

**성공 기준**:
- [ ] 목록 형식이 올바르게 적용됨
- [ ] 항목 간 들여쓰기가 정확함

---

### 4.4 뷰 타입 전환 (페이지 → 테이블)

**목적**: 문서의 뷰 타입을 페이지에서 테이블로 전환할 수 있는지 확인

**전제 조건**:
- 문서가 페이지 뷰로 열려 있음

**테스트 단계**:
1. 문서 헤더 영역 확인
2. 테이블 뷰 전환 버튼 또는 아이콘 클릭
3. (확인 다이얼로그가 있으면) 확인

**예상 결과**:
- 뷰가 테이블 뷰로 전환됨
- 문서가 테이블 형식으로 표시됨

## Edge Cases 및 오류 처리

### EC-1: 네트워크 오류 시나리오

**목적**: 네트워크 오류 시 적절한 에러 메시지가 표시되는지 확인

**테스트 단계**:
1. 개발자 도구에서 네트워크를 오프라인으로 설정
2. 문서 생성 또는 편집 시도

**예상 결과**:
- 오류 메시지가 표시됨
- 사용자가 오류 원인을 이해할 수 있음
- 애플리케이션이 크래시하지 않음

---

### EC-2: 빈 제목으로 문서 생성

**목적**: 제목 없이 문서를 생성하려 할 때 처리 확인

**테스트 단계**:
1. "새 문서" 버튼 클릭
2. 제목을 입력하지 않고 Enter

**예상 결과**:
- 기본 제목("제목 없음")이 자동으로 설정됨
- 또는 제목 입력을 요구하는 검증 메시지 표시

---

### EC-3: 매우 긴 텍스트 입력

**목적**: 제목이나 속성 값에 매우 긴 텍스트 입력 시 처리 확인

**테스트 단계**:
1. 문서 제목에 500자 이상의 긴 텍스트 입력
2. 저장 시도

**예상 결과**:
- 적절한 길이 제한이 적용됨
- 또는 UI에서 텍스트가 적절히 잘림/줄바꿈됨
- 오류가 발생하지 않음

---

### EC-4: 동시에 여러 속성 추가

**목적**: 빠르게 여러 속성을 연속으로 추가해도 오류가 발생하지 않는지 확인

**테스트 단계**:
1. "속성 추가" 버튼을 빠르게 여러 번 클릭
2. 각 속성에 이름 입력

**예상 결과**:
- 모든 속성이 정상적으로 추가됨
- UI가 일관성 있게 업데이트됨

---

### EC-5: 브라우저 뒤로 가기

**목적**: 브라우저의 뒤로 가기 버튼 사용 시 정상 동작 확인

**테스트 단계**:
1. 문서 A 열기
2. 문서 B로 이동
3. 브라우저 뒤로 가기 버튼 클릭

**예상 결과**:
- 문서 A로 정상적으로 돌아감
- 문서 A의 내용이 정확히 로드됨
